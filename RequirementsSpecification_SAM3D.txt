	•	Introduction

	•	Overview
This document defines the software requirements for the Local SAM 3D Avatar Generation System, a full-stack web application developed by a six-member team for CSCI 4050/6050, Spring 2026. This document is intended to serve both as a contract between the development team and the course instructors/stakeholders, and as a reference guide throughout the development lifecycle.
The system allows a user to upload a personal photograph and receive a rigged, textured 3D avatar (.glb format) generated by Meta's SAM 3D Body model. The avatar is then displayed interactively in a browser-based Three.js environment. The AI inference model is hosted entirely on a local machine equipped with a high-VRAM NVIDIA GPU, giving the team full control over the model and eliminating recurring cloud compute costs.
	•	Goals and Objectives
	•	Deploy Meta's SAM 3D Body model on a local GPU-equipped machine and expose it through a clean FastAPI interface.
	•	Provide a reliable and performant avatar generation pipeline from image upload to .glb file delivery.
	•	Develop an interactive Three.js web viewer capable of loading, displaying, and animating the generated 3D avatar.
	•	Implement a user authentication system backed by a local MongoDB database.
	•	Store and associate generated avatars with user accounts for persistent retrieval.
	•	Provide basic administrative capabilities to manage users and monitor system state.

	•	Scope
The system will provide end-to-end functionality for authenticated users to upload photographs, generate photorealistic 3D body avatars, and interact with those avatars in a real-time browser environment. All AI inference, file storage, and database operations will occur on a single local machine. The system will not support cloud deployment, multi-machine distribution, or real-time multiplayer interaction in the initial release. The system is scoped to serve a small number of concurrent users given the sequential nature of GPU inference.
	•	Definitions
SAM 3D Body – Meta's pre-trained model for generating a 3D human body mesh (vertices, faces, texture) from a single photograph.
Avatar – A rigged, textured 3D humanoid model in .glb format generated from a user's photograph.
.glb – GL Transmission Format Binary — a compact binary 3D model format consumable by Three.js GLTFLoader.
FastAPI – A modern Python web framework used as the backend API layer between the browser and the AI model.

Three.js – A JavaScript 3D library used to render and animate the generated avatar in the browser.
MongoDB – A local NoSQL document database used to persist user and avatar metadata.
Beanie ODM – An asynchronous Python Object Document Mapper used to abstract direct MongoDB interactions.
VRAM – Video Random Access Memory — GPU memory required to load and run the SAM 3D Body model checkpoint.
MHR Skeleton – The MakeHuman Rigged skeleton format embedded in the .glb output, enabling animation playback.
Trimesh – A Python library used to package AI model output (vertices/faces/texture) into a .glb file.
CUDA – NVIDIA's parallel computing platform required for GPU-accelerated model inference.
Conda – A Python environment manager used to isolate the AI model's complex dependency chain.
Cold Start – The initial delay (~20 seconds) incurred when loading the model checkpoint into VRAM for the first time.
Warm Model – The state of the model after the initial checkpoint load, ready for fast inference on new images.

	•	Assumptions
	•	The local machine hosting the backend has an NVIDIA GPU with at least 12 GB of VRAM.
	•	CUDA Toolkit (11.8 or 12.1), PyTorch, and Detectron2 are properly installed and configured.
	•	The SAM 3D Body model checkpoint (model.ckpt, ~1.69 GB) is available on the local machine.
	•	The application is accessed by users on the same local network or via localhost.
	•	Users upload single, front-facing photographs for best avatar generation results.
	•	MongoDB Community Edition is installed and running as a local service.

	•	General Design Constraints

	•	Product Environment
The system runs entirely on a local machine running Windows 10/11 (with WSL2 recommended) or Ubuntu. The FastAPI backend serves both the REST API and static 3D model files. The browser-based frontend communicates exclusively with this local server. No external cloud services or third-party APIs are required at runtime. The AI model is loaded once at server startup and kept warm in GPU VRAM to avoid repeated cold-start delays.
	•	User Characteristics
The system targets three user roles:
Registered User – A person with an active account who uploads photos and interacts with generated avatars. Expected to have basic computer proficiency and browser familiarity. No 3D or AI knowledge required.
Administrator – A technically proficient user (typically a developer or team member) who manages users, monitors the system, and can inspect MongoDB records directly. Accesses admin API endpoints.
Developer / Operator – A team member responsible for environment setup, model deployment, and server maintenance. Requires familiarity with Python, Conda, CUDA, and Linux/WSL2.
	•	Mandated Constraints
	•	The AI inference must use Meta's SAM 3D Body model; no alternative model may be substituted.
	•	All data (images, .glb files, user records) must be stored locally — no cloud storage.
	•	The backend must be implemented in Python using the FastAPI framework.
	•	The frontend must use React and/or Vanilla JS with Three.js for 3D rendering.
	•	MongoDB must be used as the database; no relational databases are in scope.
	•	GPU inference must be sequential (one request at a time) due to VRAM constraints.

	•	Potential System Evolution
	•	Support for batch avatar generation from multiple photographs.
	•	Integration of a richer animation library (additional poses, gestures, facial expressions).
	•	A web-based administration dashboard for managing users and monitoring the generation queue.
	•	Optional cloud deployment with a GPU-enabled cloud instance.
	•	Support for avatar customization (clothing, accessories, color palettes).
	•	Export functionality allowing users to download their .glb avatar for use in other platforms.

	•	Nonfunctional Requirements

	•	Operational Requirements
The FastAPI server must start successfully and load the SAM 3D Body model checkpoint into VRAM before accepting inference requests. The server is expected to operate continuously during a development/demo session without restart. On startup, the system must confirm GPU availability and model load status before marking itself ready.
	•	Performance Requirements
Avatar generation time (excluding cold start) should not exceed 60 seconds per image on the target hardware (NVIDIA GPU, 12+ GB VRAM). The model cold-start (checkpoint loading) is expected to take approximately 20 seconds and occurs only once per server session. API response times for non-inference endpoints (login, avatar list retrieval, static file serving) should not exceed 500 milliseconds. MongoDB queries are expected to complete within 1 second given the small user base.
	•	Security Requirements
User passwords must be hashed before storage in MongoDB; plaintext password storage is prohibited. Authentication tokens (JWT or session-based) must be required for all protected endpoints. The static file server must only serve files belonging to the authenticated requesting user. Administrator endpoints must require a separate elevated credential or role flag. As this is a local-network application, HTTPS is not mandated for the initial release but is recommended for any future external deployment.
	•	Safety Requirements
There are no direct safety requirements. However, uploaded images should be validated for file type (JPEG, PNG) and size (max 10 MB) before being passed to the AI model to prevent unexpected behavior or resource exhaustion during inference.
	•	Legal Requirements
The team has obtained access to the SAM 3D Body model weights through Meta's official Hugging Face request process. The model must be used in accordance with Meta's provided license terms and must not be redistributed. User-uploaded photographs are stored locally and are not shared with any third party.
	•	Other Quality Attributes
Maintainability – Code should follow modular design principles. The SAM3DBodyEstimator class must be treated as a black-box module and called only through its published abstract interface, not modified.
Portability – The Conda environment and its dependencies (CUDA, PyTorch, Detectron2, Trimesh) must be fully documented such that any team member with compatible hardware can reproduce the environment.
Reliability – If avatar generation fails (e.g., invalid image, model error), the API must return a structured error response and update the avatar status to 'failed' in MongoDB.

Usability – The frontend UI must allow a non-technical user to upload a photo and view their avatar within three or fewer user actions after logging in.
	•	Documentation and Training
The team will maintain internal documentation covering: (1) environment setup and dependency installation for the local GPU machine, (2) results and observations from SAM 3D Body isolation testing, and (3) API endpoint reference. No end-user training manual is required for the initial academic release, but a README covering local setup and usage will be included in the repository.
	•	External Interface
	•	User Interface
The user-facing interface is browser-based and built with React and/or Vanilla JS. It consists of:
	•	Dashboard — a grid view of the user's previously generated avatars with status indicators.
	•	Upload page — a file picker and submit button to send a photograph to the /generate endpoint.
	•	3D Viewer — a Three.js canvas displaying the generated .glb avatar with idle/walk animations and mouse-based orbit controls.
	•	Software Interface
The backend exposes a RESTful HTTP API via FastAPI. The frontend communicates with it over HTTP using JSON for metadata and multipart/form-data for file uploads. The Three.js client retrieves .glb files directly from the FastAPI static file server at a URL of the form http://localhost:8000/static/<filename>.glb. MongoDB is accessed internally by the backend only; no direct database access is exposed to the client.

	•	Use-Case: New User Registration
Actors: Unregistered user
Description: The use-case begins when a new user navigates to the registration page and submits their credentials.
Basic Path:
	•	User navigates to the registration page.
	•	User enters a username, email address, and password.
	•	System validates that the username and email are unique and that the password meets minimum requirements.
	•	System hashes the password and stores the new user record in the MongoDB users collection.
	•	System returns a success response and redirects the user to the login page.
Alternate Path:
	•	If the username or email already exists, the system returns an error indicating which field is in conflict.
	•	User is prompted to choose a different value and resubmit.
	•	Use-Case: User Login
Actors: Registered user
Description: The use-case begins when a registered user navigates to the login page.
Basic Path:
	•	System verifies the credentials against the MongoDB users collection.
	•	System issues an authentication token (JWT) and returns it to the client.
	•	Client stores the token and redirects the user to the Dashboard.
Alternate Path:
	•	If credentials are invalid, the system returns an error; the user is prompted to retry.
	•	Additional Requirements

All API routes except /register and /login must reject requests missing a valid authentication token with HTTP 401. Tokens must have a configurable expiry (default: 24 hours).

	•	Use-Case: Submit Photo for Avatar Generation
Actors: Registered user
Description: The use-case begins when an authenticated user selects a photograph and submits it to the /generate endpoint.
Basic Path:
	•	User navigates to the Upload page and selects a photograph (JPEG or PNG, max 10 MB).
	•	Frontend sends a multipart/form-data POST request to /generate with the image and the user's auth token.
	•	Backend validates the file type and size.
	•	Backend saves the image to ./storage/uploads/ and creates an avatar record in MongoDB with status 'processing'.
	•	Backend acquires the GPU inference lock and passes the image to SAM3DBodyEstimator.
	•	SAM3DBodyEstimator returns raw vertex, face, and texture data.
	•	Backend uses Trimesh to package the output into a .glb file saved to ./storage/avatars/.
	•	Backend updates the avatar MongoDB record: status  'ready', local_model_path  path to
.glb.
	•	Backend returns a JSON response containing the avatar ID and the static URL for the .glb file.
Alternate Path:
	•	If the file fails validation (wrong type or oversized), the API returns HTTP 400 with an error message; no record is created.
	•	If SAM3DBodyEstimator raises an exception, the avatar record status is set to 'failed' and the API returns HTTP 500.
	•	If the GPU lock cannot be acquired within a configurable timeout, the API returns HTTP 503 (Service Unavailable).
	•	Additional Requirements
The SAM3DBodyEstimator must be instantiated once at server startup and held in memory (warm) for the duration of the session. Re-loading the checkpoint per request is prohibited.

	•	Use-Case: View Avatar Dashboard
Actors: Registered user
Description: The use-case begins when an authenticated user navigates to the Dashboard.
Basic Path:
	•	Frontend sends a GET request to /avatars with the user's auth token.
	•	Backend queries the MongoDB avatars collection for all records belonging to the authenticated user.
	•	Backend returns a JSON list of avatar metadata (ID, status, thumbnail URL or original image URL, creation timestamp).
	•	Frontend renders each avatar as a card in a grid layout, with status badges for 'processing', 'ready', or 'failed'.
	•	Use-Case: Delete Avatar
Actors: Registered user
Basic Path:
	•	User selects an avatar on the Dashboard and clicks 'Delete'.
	•	Frontend sends a DELETE request to /avatars/{avatar_id}.
	•	Backend verifies the avatar belongs to the requesting user.
	•	Backend deletes the .glb file and original image from the filesystem.
	•	Backend removes the avatar record from MongoDB.
	•	Backend returns HTTP 204; frontend removes the card from the dashboard.
	•	Additional Requirements
Users must only be able to access and delete their own avatars. Attempting to access another user's avatar ID must result in HTTP 403.

	•	Feature: 3D Viewer and Animation
	•	Description and Priority
The 3D Viewer renders the generated avatar in the browser using Three.js. The viewer supports interactive orbit controls and applies standard animations (idle, walk) to the avatar's MHR skeleton.

	•	Use-Case: Load and View Avatar
Actors: Registered user
Description: The use-case begins when the user clicks on a 'ready' avatar card on the Dashboard.
Basic Path:
	•	User clicks the avatar card.
	•	Frontend navigates to the 3D Viewer page and passes the .glb static URL.
	•	Three.js GLTFLoader fetches the .glb file from the FastAPI static file server.
	•	The loader parses the .glb and adds the 3D mesh to the Three.js scene.
	•	The viewer applies default lighting and an idle animation clip from the MHR skeleton.
	•	The user can orbit the camera around the avatar using mouse drag or touch controls.
	•	Use-Case: Play Animation
Actors: Registered user
Basic Path:
	•	User selects an animation from a control panel (e.g., 'Idle', 'Walk').
	•	Frontend transitions the Three.js AnimationMixer to the selected animation clip.
	•	Animation plays in a loop until the user selects a different animation or closes the viewer.
	•	Additional Requirements
The viewer must function in all modern Chromium-based and Firefox browsers. The .glb file must include the MHR skeleton and all animation clips within the single file; no separate animation files should be required.
